# Phase 1 Plan 02: Database Foundation Summary

**Drizzle schema + RLS policies + audit/block-hard-delete triggers + Supabase custom access token hook + cross-tenant Vitest harness — the multi-tenant, append-only-audited Postgres core that every later phase will bolt onto.**

## Accomplishments

- Six Phase 1 tables (schools, bases, users, user_roles, documents, audit_log) with timestamptz everywhere, full Drizzle type inference, and inline `pgPolicy` declarations keyed on `auth.jwt() ->> 'school_id'`
- Generic audit trigger function `audit.fn_log_change()` (SECURITY DEFINER) that detects soft-delete transitions, reads tenant context from `app.*` GUCs, and is the only writer to `audit_log`
- `fn_block_hard_delete` BEFORE DELETE trigger and an `audit.attach(text)` helper so adding a new safety-relevant table in any future phase is one line of SQL
- Supabase custom access token hook (`public.custom_access_token_hook`) that reads `public.users` + `public.user_roles` and injects `school_id` / `roles[]` / `active_role` claims into the JWT — locked down to `supabase_auth_admin` only
- `withSchoolContext` transaction helper using `set_config(name, value, true)` so the RLS GUCs are transaction-scoped (safe under Supabase's transaction-mode pooler) and the audit trigger sees the right `user_id`
- Hand-authored `packages/db/migrations/0000_init.sql` covering enums → tables → indexes → ENABLE RLS → policies → audit schema/function → block-hard-delete → audit.attach helper → custom access token hook → trigger attachment → audit_log REVOKEs
- `supabase/config.toml` + `seed.sql` (idempotent two-school fixture) + `README.md`
- `tests/rls` Vitest workspace with `dbAsAdmin` / `asUserOf` / `seedTwoSchools` harness primitives and a `cross-tenant.test.ts` covering: JWT claim plumbing, cross-tenant SELECT/UPDATE isolation on documents/users/user_roles, audit row creation on insert + soft-delete, hard-delete prevention (P0001 raise)
